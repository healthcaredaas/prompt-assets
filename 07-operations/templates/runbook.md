## 四、日常运维

### 4.1 日常巡检

#### 4.1.1 巡检时间
- 每日 09:00 进行日常巡检
- 每周一进行周巡检
- 每月1日进行月巡检

#### 4.1.2 巡检项目

| 巡检项 | 检查内容 | 正常标准 | 异常处理 |
|--------|----------|----------|----------|
| 服务器状态 | CPU/内存/磁盘 | CPU<80%, 内存<85%, 磁盘<80% | 扩容或清理 |
| 服务状态 | 进程状态 | 运行中 | 重启服务 |
| 数据库状态 | 连接数/慢查询 | 连接数<80%, 无慢查询 | 优化或扩容 |
| 缓存状态 | 内存使用率 | <80% | 清理或扩容 |
| 日志检查 | 错误日志 | 无ERROR | 分析处理 |

#### 4.1.3 巡检脚本

```bash
#!/bin/bash
# 日常巡检脚本

echo "===== 系统巡检开始 ====="
echo "时间: $(date)"

# CPU检查
cpu_usage=$(top -bn1 | grep "Cpu(s)" | awk '{print $2}')
echo "CPU使用率: ${cpu_usage}%"

# 内存检查
mem_usage=$(free | grep Mem | awk '{print $3/$2 * 100}')
echo "内存使用率: ${mem_usage}%"

# 磁盘检查
disk_usage=$(df -h | grep -E "^/dev" | awk '{print $5}')
echo "磁盘使用率: ${disk_usage}"

# 服务检查
systemctl status app-service | grep Active

echo "===== 系统巡检结束 ====="
```

## 六、故障处理

### 6.1 故障分级

| 级别 | 定义 | 响应时间 | 处理时间 |
|------|------|----------|----------|
| P0 | 核心功能不可用，影响全部用户 | 5分钟 | 30分钟 |
| P1 | 主要功能不可用，影响大量用户 | 15分钟 | 1小时 |
| P2 | 次要功能异常，影响部分用户 | 30分钟 | 4小时 |
| P3 | 功能异常，影响少数用户 | 2小时 | 24小时 |

### 6.2 常见故障处理

#### 6.2.1 服务无法启动

**现象**: 服务启动失败

**排查步骤**:
1. 检查日志: `journalctl -u app-service -n 100`
2. 检查端口: `netstat -tlnp | grep 8080`
3. 检查配置: 检查配置文件是否正确
4. 检查依赖: 检查数据库、缓存等依赖服务

**解决方案**:
```bash
# 检查服务状态
systemctl status app-service

# 查看详细日志
journalctl -u app-service -n 200

# 检查配置文件
cat /etc/app/application.yml

# 重启依赖服务
systemctl restart mysql redis
```

#### 6.2.2 服务响应慢

**现象**: 服务响应时间变长

**排查步骤**:
1. 检查系统资源: CPU、内存、磁盘IO
2. 检查数据库: 慢查询、连接数
3. 检查缓存: 命中率、内存使用
4. 检查网络: 带宽、延迟

**解决方案**:
```bash
# 检查系统资源
top -c
iostat -x 1

# 检查数据库慢查询
mysql -e "SHOW PROCESSLIST"

# 检查缓存命中率
redis-cli info stats | grep hit_rate

# 分析应用性能
curl http://localhost:8080/actuator/metrics
```

#### 6.2.3 数据库连接异常

**现象**: 应用报数据库连接异常

**排查步骤**:
1. 检查数据库状态
2. 检查连接数
3. 检查网络连通性
4. 检查用户权限

**解决方案**:
```bash
# 检查数据库状态
systemctl status mysql

# 检查连接数
mysql -e "SHOW STATUS LIKE 'Threads_connected'"
mysql -e "SHOW VARIABLES LIKE 'max_connections'"

# 检查网络
telnet db-host 3306

# 重启数据库
systemctl restart mysql
```

## 八、备份恢复

### 8.1 备份策略

| 备份对象 | 备份类型 | 备份频率 | 保留时间 | 存储位置 |
|----------|----------|----------|----------|----------|
| 数据库 | 全量 | 每日 02:00 | 30天 | 备份服务器 |
| 数据库 | 增量 | 每小时 | 7天 | 备份服务器 |
| 配置文件 | 全量 | 每日 02:00 | 90天 | Git仓库 |
| 应用日志 | 增量 | 每日 03:00 | 30天 | 日志服务器 |

### 8.2 备份脚本

```bash
#!/bin/bash
# 数据库备份脚本

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backup/mysql"
DB_NAME="app_db"
MYSQL_USER="backup"
MYSQL_PASS="password"

# 创建备份目录
mkdir -p ${BACKUP_DIR}

# 全量备份
mysqldump -u${MYSQL_USER} -p${MYSQL_PASS} \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  ${DB_NAME} | gzip > ${BACKUP_DIR}/${DB_NAME}_${DATE}.sql.gz

# 删除7天前的备份
find ${BACKUP_DIR} -name "*.sql.gz" -mtime +7 -delete

echo "Backup completed: ${BACKUP_DIR}/${DB_NAME}_${DATE}.sql.gz"
```

### 8.3 恢复流程

```bash
#!/bin/bash
# 数据库恢复脚本

BACKUP_FILE="/backup/mysql/app_db_20240101_020000.sql.gz"
DB_NAME="app_db"
MYSQL_USER="root"
MYSQL_PASS="password"

# 停止应用
systemctl stop app-service

# 恢复数据库
gunzip < ${BACKUP_FILE} | mysql -u${MYSQL_USER} -p${MYSQL_PASS} ${DB_NAME}

# 启动应用
systemctl start app-service

echo "Restore completed"
```

## 九、应急预案

### 9.1 服务器故障

**场景**: 服务器硬件故障

**应急步骤**:
1. 确认故障服务器
2. 切换流量到备用服务器
3. 联系硬件供应商
4. 更换故障硬件
5. 恢复服务

### 9.2 数据库故障

**场景**: 数据库主节点故障

**应急步骤**:
1. 确认主节点状态
2. 提升从节点为主节点
3. 修改应用连接配置
4. 重启应用服务
5. 重建从节点

### 9.3 网络故障

**场景**: 网络中断

**应急步骤**:
1. 确认故障范围
2. 切换备用网络
3. 联系网络运营商
4. 恢复网络连接
5. 验证服务状态
