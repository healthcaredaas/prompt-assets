# Prompt 语法指南

本文档定义了 Prompt 资产库中所有 Prompt 的统一语法规范。

## Prompt 结构规范

### 基本结构

每个 Prompt 文件应包含以下部分：

```markdown
# [Prompt名称]

## 元信息
- 版本: v1.0
- 适用角色: [产品/开发/测试/运维]
- 输入要求: [需要的上下文信息]
- 输出格式: [期望的输出格式]
- 依赖模板: [可选，引用的模板文件]

## 角色设定
你是一个[具体角色]，负责[具体职责]...

## 上下文
[项目背景、技术栈、约束条件]

## 任务描述
[具体的任务说明]

## 输出要求
1. [输出格式要求]
2. [质量标准]
3. [交付物清单]

## 示例
### 输入
[示例输入]

### 输出
[示例输出]

## 变量说明
- {{project_name}}: 项目名称
- {{tech_stack}}: 技术栈
- {{constraints}}: 约束条件
```

---

## 元信息字段说明

### 版本 (Version)
采用语义化版本：`vMAJOR.MINOR.PATCH`

| 字段 | 说明 |
|------|------|
| MAJOR | 不兼容的重大变更 |
| MINOR | 向后兼容的功能新增 |
| PATCH | 向后兼容的问题修复 |

### 适用角色 (Target Role)
| 角色 | 说明 |
|------|------|
| 产品经理 | 需求分析、产品规划 |
| 需求分析师 | 需求细化、文档编写 |
| UI设计师 | 界面设计、交互设计 |
| 架构师 | 技术架构、系统设计 |
| 后端开发 | 服务端开发、API实现 |
| 前端开发 | 客户端开发、UI实现 |
| 测试工程师 | 测试设计、质量保障 |
| DevOps工程师 | 部署运维、CI/CD |
| 运维工程师 | 系统运维、故障处理 |
| 安全工程师 | 安全审计、漏洞修复 |

### 输入要求 (Input Requirements)
描述使用该 Prompt 时需要提供的信息：
- 必需信息：必须提供的上下文
- 可选信息：可选的补充说明
- 格式要求：数据格式规范

### 输出格式 (Output Format)
定义期望的输出形式：
- 文档格式：Markdown、JSON、YAML
- 代码格式：Java、Vue、SQL、YAML
- 图表格式：Mermaid、PlantUML

## 变量系统

### 变量定义

使用双大括号语法：`{{variable_name}}`

```markdown
## 上下文
- 项目名称: {{project_name}}
- 技术栈: {{tech_stack}}
- 部署环境: {{deployment_env}}
```

### 内置变量

| 变量名 | 说明 | 示例值 |
|--------|------|--------|
| `{{project_name}}` | 项目名称 | 订单管理系统 |
| `{{tech_stack}}` | 技术栈 | Java + Spring Cloud |
| `{{deployment_env}}` | 部署环境 | Kubernetes |
| `{{team_size}}` | 团队规模 | 10人 |
| `{{sprint_duration}}` | 迭代周期 | 2周 |
| `{{code_style}}` | 代码规范 | 阿里巴巴Java开发手册 |

### 自定义变量

根据具体需求定义：
```markdown
## 变量说明
- {{api_version}}: API版本号，如 v1.0
- {{service_name}}: 微服务名称
- {{database_type}}: 数据库类型
```

## 条件逻辑

### 条件判断

```markdown
{{#if condition}}
  内容A（条件为真时显示）
{{else}}
  内容B（条件为假时显示）
{{/if}}
```

### 循环遍历

```markdown
{{#each items}}
- {{this.name}}: {{this.description}}
{{/each}}
```

## 质量标准

### Prompt 编写原则

1. **明确性**: 任务描述清晰无歧义
2. **完整性**: 包含所有必要信息
3. **可执行性**: 输出可验证、可交付
4. **复用性**: 适配多个相似场景
5. **可维护性**: 结构清晰、易于更新

### 输出质量要求

1. **准确性**: 输出内容正确无误
2. **完整性**: 覆盖所有要求的输出项
3. **规范性**: 符合行业标准和团队规范
4. **实用性**: 可直接用于实际工作
5. **可追溯**: 来源可查、依据可溯

## 示例模板

### 最小可用 Prompt

```markdown
# 用户故事生成

## 元信息
- 版本: v1.0
- 适用角色: 产品经理
- 输入要求: 功能需求描述
- 输出格式: 用户故事列表

## 角色设定
你是一个敏捷开发专家，擅长将功能需求转化为用户故事。

## 任务描述
将输入的功能需求分解为标准格式的用户故事。

## 输出要求
输出用户故事列表，每个故事包含：
- 作为[角色]
- 我想要[功能]
- 以便于[价值]
- 验收标准

## 示例
输入: 用户需要登录功能
输出:
- 作为注册用户，我想要使用账号密码登录，以便于访问系统。
  验收标准：
  1. 支持用户名/邮箱登录
  2. 密码错误提示
  3. 登录失败次数限制
```
