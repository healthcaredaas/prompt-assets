## 上下文

### 项目背景
- **技术栈**: Java + Spring Cloud 后端, Vue 前端
- **部署环境**: Docker + Kubernetes
- **集成方式**: REST API, RPC, 消息队列
- **安全等级**: 企业级

### 集成测试原则
1. **接口优先**: 重点测试服务间接口
2. **数据一致**: 验证数据流转一致性
3. **异常处理**: 覆盖集成异常场景
4. **性能验证**: 验证集成性能指标

## 四、集成测试用例

### 4.1 接口测试用例

#### TC-INT-001: 库存预占接口测试
**接口信息**
- 接口地址: /api/inventory/reserve
- 请求方式: POST
- 提供方: 库存服务
- 调用方: 订单服务

**测试步骤**
| 步骤 | 操作 | 请求数据 | 预期响应 |
|------|------|----------|----------|
| 1 | 查询当前库存 | productId=1001 | stock=100 |
| 2 | 调用预占接口 | productId=1001, qty=2 | success, reserveId=xxx |
| 3 | 再次查询库存 | productId=1001 | availableStock=98 |
| 4 | 预占超时未支付 | 等待30分钟 | 预占自动释放 |
| 5 | 查询库存 | productId=1001 | stock=100 |

**数据验证**
- 请求验证: productId和qty参数正确
- 响应验证: reserveId不为空，状态为RESERVED
- 数据库验证: 库存记录表有预占记录

### 4.2 数据一致性测试

#### TC-DATA-001: 支付成功数据一致性测试
**测试目的**: 验证支付成功后各服务数据一致性

**测试步骤**
1. 创建订单，库存预占成功
2. 发起支付，获取支付链接
3. 模拟支付成功
4. 验证消息消费成功
5. 验证各服务数据状态

**验证点**
- 订单服务数据: 订单状态=PAID, 支付时间不为空
- 库存服务数据: 库存扣减成功，预占记录=DEDUCTED
- 通知服务数据: 通知记录已生成

### 4.3 异常处理测试

#### TC-ERR-001: 库存服务超时测试
**异常模拟**: 使用ChaosBlade注入网络延迟

**测试步骤**
1. 注入库存服务网络延迟5秒
2. 创建订单
3. 验证重试机制
4. 验证最终结果

**预期结果**
- 异常捕获: 订单服务捕获超时异常
- 重试机制: 重试3次，间隔1秒
- 降级处理: 3次失败后返回库存服务不可用
- 日志记录: 记录超时日志和重试次数

#### TC-ERR-002: 分布式事务一致性测试
**异常模拟**: 在库存扣减后模拟订单服务宕机

**测试步骤**
1. 创建订单，库存预占成功
2. 支付成功，发送消息
3. 在消息消费时模拟服务重启
4. 验证消息重新消费
5. 验证最终一致性

**预期结果**
- 消息不丢失: 消息队列持久化
- 幂等处理: 重复消费不会重复扣减
- 最终一致: 30秒内达到一致状态
