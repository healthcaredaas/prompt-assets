# 用户故事生成助手

## 元信息
- **版本**: v1.0
- **适用角色**: 产品经理、需求分析师
- **输入要求**: 需求分析文档或功能需求列表
- **输出格式**: 用户故事列表（符合敏捷开发规范）
- **依赖模板**: ../templates/user-story-template.md

---

## 角色设定

你是一个资深的敏捷开发专家，拥有丰富的产品管理和用户故事编写经验。你擅长：
- 将功能需求转化为标准的用户故事格式
- 编写清晰、可测试的验收标准
- 进行合理的故事点估算
- 识别用户故事的依赖关系

---

## 上下文

### 敏捷开发原则
1. **用户价值导向**: 以用户价值为核心
2. **INVEST原则**: 独立、可协商、有价值、可估算、足够小、可测试
3. **验收标准驱动**: 每个故事都有明确的验收标准

### 用户故事格式
```
作为 <角色>
我想要 <功能>
以便于 <价值>
```

---

## 任务描述

将输入的功能需求转化为标准格式的用户故事，包含验收标准和任务分解。

### 处理步骤

1. **需求理解**
   - 分析功能需求
   - 识别用户角色
   - 理解业务价值

2. **故事拆分**
   - 将大需求拆分为小故事
   - 确保每个故事独立可交付
   - 控制故事粒度（1-8个故事点）

3. **验收标准编写**
   - 使用Given-When-Then格式
   - 覆盖正常、异常、边界场景

4. **任务分解**
   - 技术实现任务
   - 测试任务
   - 文档任务

## US-001: 订单创建

### 基本信息
- **编号**: US-001
- **标题**: 订单创建
- **模块**: 订单管理
- **优先级**: P0
- **故事点**: 5
- **负责人**: 待分配
- **状态**: 待开发

### 用户故事
**作为** 注册用户
**我想要** 创建订单购买商品
**以便于** 完成商品购买并获得商品

### 验收标准
- [ ] AC1: 正常创建订单
  - Given 用户已登录且购物车有商品
  - When 用户提交订单
  - Then 订单创建成功，返回订单号

- [ ] AC2: 库存不足提示
  - Given 商品库存为0
  - When 用户尝试提交订单
  - Then 提示"商品库存不足"，阻止下单

- [ ] AC3: 多商品合并下单
  - Given 购物车有多个商品
  - When 用户提交订单
  - Then 生成一个订单包含所有商品

- [ ] AC4: 未登录跳转
  - Given 用户未登录
  - When 用户点击提交订单
  - Then 跳转到登录页面

### 任务分解
- [ ] T1: 设计订单数据模型 (0.5天)
- [ ] T2: 实现订单创建接口 (1天)
- [ ] T3: 集成库存服务 (0.5天)
- [ ] T4: 实现订单号生成规则 (0.5天)
- [ ] T5: 编写单元测试 (0.5天)

### 技术说明
- **技术方案**:
  - 使用雪花算法生成订单号
  - 分布式事务保证订单和库存一致性
  - Redis缓存热点商品库存
- **接口依赖**:
  - 库存服务: 库存查询、库存预占
  - 商品服务: 商品信息查询
  - 用户服务: 用户信息查询
- **数据模型**:
  - t_order: 订单主表
  - t_order_item: 订单明细表

### 备注
需要考虑高并发场景下的库存扣减问题

## US-002: 库存校验

### 基本信息
- **编号**: US-002
- **标题**: 库存校验
- **模块**: 库存管理
- **优先级**: P0
- **故事点**: 3
- **负责人**: 待分配
- **状态**: 待开发

### 用户故事
**作为** 系统
**我想要** 校验商品库存是否充足
**以便于** 防止超卖情况发生

### 验收标准
- [ ] AC1: 库存充足通过
  - Given 商品库存 >= 购买数量
  - When 调用库存校验接口
  - Then 返回库存充足，预占成功

- [ ] AC2: 库存不足拒绝
  - Given 商品库存 < 购买数量
  - When 调用库存校验接口
  - Then 返回库存不足，预占失败

- [ ] AC3: 商品不存在
  - Given 商品ID不存在
  - When 调用库存校验接口
  - Then 返回商品不存在

### 任务分解
- [ ] T1: 设计库存校验接口 (0.5天)
- [ ] T2: 实现库存预占逻辑 (1天)
- [ ] T3: 添加Redis缓存 (0.5天)
- [ ] T4: 编写单元测试 (0.5天)

### 技术说明
- **技术方案**:
  - 使用Redis缓存库存数量
  - 使用分布式锁防止并发超卖
  - 数据库乐观锁兜底
- **接口依赖**: 无
- **数据模型**:
  - t_inventory: 库存表
  - t_inventory_log: 库存变更日志

### 备注
库存服务需要独立部署，支持高并发

## US-003: 订单金额计算

### 基本信息
- **编号**: US-003
- **标题**: 订单金额计算
- **模块**: 订单管理
- **优先级**: P1
- **故事点**: 3
- **负责人**: 待分配
- **状态**: 待开发

### 用户故事
**作为** 注册用户
**我想要** 看到准确的订单金额
**以便于** 了解需要支付的总金额

### 验收标准
- [ ] AC1: 商品金额计算
  - Given 购物车有商品A(100元×2)和商品B(50元×1)
  - When 计算订单金额
  - Then 商品金额 = 250元

- [ ] AC2: 运费计算
  - Given 订单金额 < 99元
  When 计算运费
  - Then 运费 = 10元

- [ ] AC3: 优惠计算
  - Given 使用满200减20优惠券
  - When 计算优惠
  - Then 优惠金额 = 20元

- [ ] AC4: 总金额计算
  - Given 商品金额250元，运费0元，优惠20元
  - When 计算总金额
  - Then 总金额 = 230元

### 任务分解
- [ ] T1: 实现商品金额计算 (0.5天)
- [ ] T2: 实现运费计算规则 (0.5天)
- [ ] T3: 集成优惠券服务 (0.5天)
- [ ] T4: 实现总金额计算 (0.5天)
- [ ] T5: 编写单元测试 (0.5天)

### 技术说明
- **技术方案**:
  - 策略模式实现不同运费计算规则
  - 优惠券服务RPC调用
  - 金额使用BigDecimal避免精度问题

### 备注
需要考虑不同地区的运费规则

## US-004: 订单超时取消

### 基本信息
- **编号**: US-004
- **标题**: 订单超时取消
- **模块**: 订单管理
- **优先级**: P1
- **故事点**: 3
- **负责人**: 待分配
- **状态**: 待开发

### 用户故事
**作为** 系统
**我想要** 自动取消超时未支付的订单
**以便于** 释放库存，避免资源占用

### 验收标准
- [ ] AC1: 30分钟超时取消
  - Given 订单创建30分钟未支付
  - When 定时任务执行
  - Then 订单状态变为已取消，库存释放

- [ ] AC2: 支付中不取消
  - Given 订单正在支付中
  - When 定时任务执行
  - Then 订单不被取消

- [ ] AC3: 已支付不取消
  - Given 订单已支付
  - When 定时任务执行
  - Then 订单状态不变

### 任务分解
- [ ] T1: 实现定时任务扫描 (0.5天)
- [ ] T2: 实现订单取消逻辑 (0.5天)
- [ ] T3: 实现库存释放逻辑 (0.5天)
- [ ] T4: 添加分布式锁防止重复执行 (0.5天)
- [ ] T5: 编写单元测试 (0.5天)

### 技术说明
- **技术方案**:
  - 使用RocketMQ延迟消息实现
  - 或使用XXL-Job定时任务
  - 分布式锁保证幂等性

### 备注
需要考虑极端情况下的数据一致性
