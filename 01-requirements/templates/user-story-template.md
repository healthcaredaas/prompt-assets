# 用户故事模板

> 版本: v1.0 | 最后更新: 2026-02-26

本文档定义用户故事的标准格式和编写规范。

---

## 用户故事格式

### 标准格式

```
作为 <角色>
我想要 <功能>
以便于 <价值>
```

### 完整格式

```markdown
## US-{{编号}}: {{故事标题}}

### 基本信息
- **编号**: US-{{编号}}
- **标题**: {{故事标题}}
- **模块**: {{所属模块}}
- **优先级**: {{P0/P1/P2/P3}}
- **故事点**: {{估算点数}}
- **负责人**: {{开发负责人}}
- **状态**: {{待开发/开发中/测试中/已完成}}

### 用户故事
**作为** {{角色}}
**我想要** {{功能描述}}
**以便于** {{业务价值}}

### 验收标准
- [ ] AC1: {{验收标准1}}
- [ ] AC2: {{验收标准2}}
- [ ] AC3: {{验收标准3}}

### 技术说明
- **技术方案**: {{技术实现方案}}
- **接口依赖**: {{依赖的接口或服务}}
- **数据模型**: {{涉及的数据表}}

### 任务分解
- [ ] T1: {{任务1}}
- [ ] T2: {{任务2}}
- [ ] T3: {{任务3}}

### 备注
{{补充说明}}
```

## 编写规范

### 角色定义

| 角色类型 | 说明 | 示例 |
|----------|------|------|
| 最终用户 | 系统的直接使用者 | 注册用户、访客 |
| 业务人员 | 执行业务操作的人员 | 客服、运营、管理员 |
| 系统管理员 | 负责系统维护的人员 | 超级管理员 |
| 外部系统 | 集成的外部系统 | 支付系统、物流系统 |

### 功能描述规范

**应该做什么**：
- ✅ 使用主动语态
- ✅ 描述用户行为
- ✅ 明确操作目标

**不应该做什么**：
- ❌ 使用技术术语
- ❌ 包含实现细节
- ❌ 模糊不清的描述

**示例对比**：

| 差的描述 | 好的描述 |
|----------|----------|
| 系统应该有一个登录接口 | 作为注册用户，我想要使用账号密码登录系统 |
| 数据库要存用户信息 | 作为管理员，我想要添加新的用户账户 |
| 前端要做个列表页 | 作为运营人员，我想要查看所有订单列表 |

### 价值描述规范

**价值维度**：
1. **效率提升**: 节省时间、提高工作效率
2. **成本降低**: 减少人力成本、运营成本
3. **体验优化**: 提升用户满意度、降低操作难度
4. **风险控制**: 避免错误、保障安全
5. **收入增长**: 增加收入来源、提升转化率

## 验收标准编写

### 格式规范

使用 Given-When-Then 格式：

```
Given [前置条件]
When [触发动作]
Then [预期结果]
```

### 示例

```markdown
### 验收标准

**AC1: 正常登录**
- Given 用户已注册且账号状态正常
- When 用户输入正确的账号和密码并点击登录
- Then 系统验证通过，跳转到首页

**AC2: 密码错误**
- Given 用户已注册
- When 用户输入错误的密码
- Then 系统提示"账号或密码错误"
- And 登录失败次数+1

**AC3: 账号锁定**
- Given 用户连续5次输入错误密码
- When 用户再次尝试登录
- Then 系统提示"账号已锁定，请30分钟后再试"
```

## 故事点估算

### 斐波那契数列

使用斐波那契数列估算：1, 2, 3, 5, 8, 13, 21

| 点数 | 复杂度 | 说明 |
|------|--------|------|
| 1 | 极简 | 几分钟完成，如文案修改 |
| 2 | 简单 | 1-2小时完成，如简单UI调整 |
| 3 | 较简单 | 半天完成，如简单CRUD |
| 5 | 中等 | 1天完成，如复杂业务逻辑 |
| 8 | 较复杂 | 2-3天完成，如集成开发 |
| 13 | 复杂 | 3-5天完成，需拆分 |
| 21 | 极复杂 | 超过5天，必须拆分 |

### 估算维度

1. **复杂度**: 逻辑复杂程度
2. **不确定性**: 需求明确程度
3. **工作量**: 工时预估

## 优先级定义

| 优先级 | 说明 | 示例 |
|--------|------|------|
| P0 | 必须完成，核心功能 | 用户登录、订单创建 |
| P1 | 重要功能，影响体验 | 搜索优化、数据导出 |
| P2 | 优化功能，提升体验 | 快捷操作、提示优化 |
| P3 | 可选功能，锦上添花 | 主题切换、动画效果 |

## 示例

### US-001: 用户登录

```markdown
## US-001: 用户登录

### 基本信息
- **编号**: US-001
- **标题**: 用户登录
- **模块**: 用户中心
- **优先级**: P0
- **故事点**: 5
- **负责人**: 张三
- **状态**: 待开发

### 用户故事
**作为** 注册用户
**我想要** 使用账号密码登录系统
**以便于** 访问系统功能进行业务操作

### 验收标准
- [ ] AC1: 支持用户名/邮箱登录
- [ ] AC2: 密码错误提示"账号或密码错误"
- [ ] AC3: 连续5次错误锁定账号30分钟
- [ ] AC4: 登录成功后跳转到首页
- [ ] AC5: 记住登录状态7天

### 技术说明
- **技术方案**: JWT Token认证
- **接口依赖**: 用户服务、权限服务
- **数据模型**: t_user, t_login_log

### 任务分解
- [ ] T1: 设计登录页面UI
- [ ] T2: 实现登录API接口
- [ ] T3: 实现Token认证中间件
- [ ] T4: 实现登录失败限制
- [ ] T5: 编写单元测试

### 备注
需要对接统一认证中心
```

### US-002: 订单创建

```markdown
## US-002: 订单创建

### 基本信息
- **编号**: US-002
- **标题**: 订单创建
- **模块**: 订单管理
- **优先级**: P0
- **故事点**: 8
- **负责人**: 李四
- **状态**: 待开发

### 用户故事
**作为** 注册用户
**我想要** 创建订单购买商品
**以便于** 完成商品购买并获得商品

### 验收标准
- [ ] AC1: 支持多商品合并下单
- [ ] AC2: 自动计算订单金额（商品+运费-优惠）
- [ ] AC3: 库存不足时提示并阻止下单
- [ ] AC4: 订单创建后库存预占
- [ ] AC5: 30分钟未支付自动取消

### 技术说明
- **技术方案**: 分布式事务（Seata）
- **接口依赖**: 库存服务、优惠服务、支付服务
- **数据模型**: t_order, t_order_item, t_order_log

### 任务分解
- [ ] T1: 设计订单数据模型
- [ ] T2: 实现订单创建接口
- [ ] T3: 集成库存服务
- [ ] T4: 集成优惠服务
- [ ] T5: 实现订单超时取消
- [ ] T6: 编写集成测试

### 备注
需要考虑高并发场景
```
